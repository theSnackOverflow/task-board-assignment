# 설계 결정 (DECISIONS)

## 1. 상태 구조

서버가 확인해 준 상태(`serverCache: ids + byId`)와 아직 확인받지 못한 내 변경(`pendingQueue: Mutation[]`)을 분리했습니다. 화면은 항상 `applyPending(serverCache, queue)`의 파생 계산입니다 (`src/lib/mutations.ts`, `src/store/taskStore.ts`).

- serverCache는 서버 응답으로만 갱신됩니다. 낙관 반영은 큐에만 존재합니다.
- 관리 위치는 React 밖의 vanilla store이고 컴포넌트는 `useSyncExternalStore`로 구독합니다. 비동기 엔진이 React 생명주기와 무관하게 큐를 처리해야 하고, StrictMode 이중 마운트에도 엔진과 큐가 1개로 유지되어야 하기 때문입니다.
- 버린 대안 1 — 단일 `tasks` 배열(스타터 방식): 서버 상태와 낙관 상태가 섞여서 실패했을 때 무엇으로 되돌릴지가 모호해집니다.
- 버린 대안 2 — TanStack Query: 실무에서 써 본 도구지만, 이 과제의 핵심인 "같은 태스크 연속 뮤테이션"에서 onMutate 스냅샷 롤백이 서로를 덮는 문제가 있어 결국 태스크별 큐를 직접 설계해야 합니다. 그 큐가 과제의 본질이라면 라이브러리 없이 순수 함수로 만들어 테스트하는 쪽이 요구사항(핵심 로직 유닛 테스트)과 정합한다고 판단했습니다.

## 2. 낙관적 업데이트 & 롤백

모든 조작(이동/수정/삭제/생성)은 큐에 넣는 즉시 화면에 반영됩니다. 실패가 확정되면 해당 뮤테이션을 큐에서 제거합니다. 파생 뷰가 재계산되면서 화면이 저절로 이전 상태가 됩니다.

"서버 재조회 vs 이전 스냅샷 복원" 중 어느 쪽인가에 대한 답은 둘 다 아니다입니다.

- 재조회 기각: 실패율 15%에서 실패마다 5,000개 GET이 추가되고, 그 GET도 2% 확률로 실패합니다.
- 스냅샷 복원 기각: 카드 A와 카드 B가 동시에 낙관 반영 중일 때 A가 실패해 A 시점 스냅샷을 복원하면 B의 반영까지 함께 사라집니다. 동시 뮤테이션이 겹칠수록 복원 순서 조합이 폭발합니다.
- 파생 뷰 방식은 롤백 코드 자체가 없어서 롤백 버그도 존재할 수 없습니다. 실패한 항목만 빠지고 나머지 pending은 유지됩니다 (테스트: `src/store/engine.test.ts`의 롤백 케이스).

실패 알림은 태스크 제목을 포함한 sticky 토스트와 "다시 시도" 액션으로 제공합니다.

## 3. 경쟁 상태 (race condition)

세 겹으로 처리했습니다 (`src/store/engine.ts`).

1. 태스크별 직렬화: 같은 태스크에 대한 요청은 동시에 1개만 나갑니다. 같은 태스크의 응답 순서 역전이 구조적으로 불가능합니다. 다른 태스크끼리는 병렬입니다.
2. version 체이닝: 뮤테이션에 version을 저장하지 않고, 엔진이 전송 직전에 serverCache에서 읽어 주입합니다. 직렬화 덕분에 이전 요청의 성공 응답(version+1)이 캐시에 반영된 뒤에야 다음 요청이 나가므로 version이 자동으로 이어집니다.
3. coalescing: 대기 중(미전송)인 같은 종류의 뮤테이션은 마지막 것으로 교체합니다. A에서 B, C, D로 연속 이동하면 요청은 2개(진행 중 1 + 대기 1)로 수렴하고 최종 상태는 D입니다.

- 버린 대안 — AbortController로 이전 요청 취소: 취소 시점에 서버가 이미 처리했는지 알 수 없습니다. 취소했다고 믿은 요청이 서버에 반영되면 클라이언트는 그 사실을 모릅니다.
- 버린 대안 — 응답 타임스탬프 비교(last-response-wins): 동시 in-flight를 허용하는 한 서버 처리 순서를 클라이언트가 통제할 수 없습니다.

move는 `{status}`만, edit는 콘텐츠 필드만 보내도록 payload를 분리해서(필드 disjoint) coalescing으로 큐 순서가 바뀌어도 서버 최종 상태가 동일하도록 만들었습니다.

## 4. 대량 데이터 성능 (5,000개)

- 선택: 컬럼별 가상 스크롤 직접 구현. `computeWindow(scrollTop, viewportHeight, rowHeight, count, overscan, forcedIndex)` 순수 함수가 렌더 범위를 계산하고 카드는 translateY로 절대 배치합니다 (`src/lib/windowing.ts`, `src/components/Column.tsx`).
- 행 높이 고정을 위해 카드 제목을 1줄 말줄임 처리했습니다. 포기한 것: 제목 여러 줄 표시(전체 제목은 툴팁과 편집 모달에서 확인), 가변 높이 가상화, 컬럼 내 위치 지정 드롭.
- 라이브러리(react-window 등)를 안 쓴 이유: 고정 행높이 + 컬럼 단위 드롭이라는 이 과제의 조건에서는 필요한 것이 순수 함수 하나 분량이고, 그 함수는 유닛 테스트로 증명할 수 있습니다.
- 드래그 중 스크롤로 원본 카드가 렌더 범위를 벗어나 unmount되면 브라우저가 드래그를 취소하는 문제가 있어, 드래그 중인 카드 인덱스를 범위에 강제 포함(forcedIndex)했습니다.
- 검색은 디바운스(연산 빈도 축소)와 useDeferredValue(갱신 우선순위 하향)를 함께 썼습니다. 5,000개 필터 연산 자체는 밀리초 단위라 병목이 아니고, DOM 렌더가 병목이므로 가상 스크롤이 본질입니다.

측정 (macOS Chrome 149, 동일 시드 5,000개, 스타터 배포 데모와 본인 배포 모두 GitHub Pages 프로덕션 빌드로 동등 조건):

| 지표 | 스타터 | 가상 스크롤 적용 |
| ---- | ---- | ---- |
| 카드 DOM 수 | 5,000 | 39 (뷰포트 + overscan) |
| 전체 DOM 노드 | 25,028 | 406 |
| 강제 리플로우 (display 토글 후 offsetHeight 재계산) | 80.1ms | 3.0ms |

## 5. 정답이 없던 결정들

### 네트워크 완전 단절

- 감지: `offline` 이벤트와 fetch의 TypeError(HTTP 응답 자체가 없음 — 500 응답과 구분)를 함께 사용합니다. `navigator.onLine`은 위양성이 있어 TypeError를 최종 판정자로 둡니다.
- 동작: 상단 배너 표시 + 신규 쓰기는 즉시 차단(토스트) + 이미 큐에 있던 작업은 재시도 예산을 소모하지 않고 보존했다가 `online` 이벤트에 자동 재개. 재개 후 큐가 비면 전체 재조회로 동기화합니다.
- 이유: "이미 한 조작은 지키고, 새 발산은 막는다". 오프라인 중에도 계속 쓰게 하면(오프라인 큐잉) 재연결 때 충돌이 몰려오고, 진행 중이던 것까지 버리면(완전 차단) 사용자가 한 일이 사라집니다. 차단+재개가 유실 없음과 괴리 유한함을 동시에 지키는 지점입니다.
- 명시하는 한계: TypeError 시 요청이 서버에 도달했는지는 원리상 알 수 없습니다. 이 과제의 mock 서버는 실패 판정이 본문 처리보다 먼저 실행되어 실패한 쓰기가 상태를 바꾸지 않으므로(핸들러 코드로 확인) 재전송이 안전하지만, 실서버라면 멱등 키 같은 장치가 필요합니다.

### 409 충돌

- 동작: 응답의 `current`(서버 최신)를 즉시 화면에 반영 + "다른 곳에서 먼저 수정됨" 토스트 + "내 변경 다시 적용" 버튼. 같은 태스크에서 3회 반복되면 버튼 없이 안내만 합니다.
- 이유: 서버 우선이라 데이터 유실이 없고, 재적용 여부는 사용자가 결정합니다. 자동 재적용은 상대의 변경을 소리 없이 덮고(두 탭이면 핑퐁이 됩니다), 비교 다이얼로그는 status 이동 충돌에는 과잉이라 기각했습니다.
- 재적용은 저장해 둔 version이 아니라 그 시점 serverCache의 version으로 전송되므로(3절의 version 주입 규칙) 구식 version에 의한 충돌 루프가 생기지 않습니다.

### 실패한 쓰기 재시도: 자동 vs 수동

- 동작: 자동 2회(300ms, 900ms + jitter 20%, 지수 백오프) 후 수동 전환(토스트의 "다시 시도").
- 이유: 실패율 15%를 전부 수동으로 넘기면 사용자가 대략 7번에 1번 클릭해야 합니다. 반대로 무한 자동은 서버 장애를 숨깁니다. 자동 2회면 3연속 실패 확률이 약 0.34%(0.15^3)로 줄고, 그 이상은 사용자가 알아야 할 상황이라고 판단했습니다.
- 재시도가 안전한 근거: 이 mock 서버는 실패 판정이 본문 처리 앞에 있어 실패한 POST가 태스크를 만들지 않습니다. 따라서 재시도로 중복 생성이 발생하지 않습니다.

## 6. 그 외 트레이드오프 / 남은 한계

- 생성 확정 시 tempId가 서버 id로 바뀌며 카드 1장이 리마운트됩니다(React key 변경). id 별칭 맵으로 피할 수 있지만 생성 직후 1회뿐이라 수용했습니다.
- jsdom 환경 한계 3건을 테스트 셋업으로 대응했습니다: dialog showModal 미구현(최소 폴리필), ResizeObserver 부재(no-op), Node BroadcastChannel과 jsdom의 이벤트 클래스 충돌(채널 계약만 구현한 테스트 대체물). 실브라우저 동작은 배포 URL에서 수동 검증했습니다.
- 키보드 이동 직후 대상 카드가 가상 스크롤 범위 밖이면 포커스 복원이 실패할 수 있습니다.
- 시간이 더 있었다면: 재연결 flush에서 실패가 몰릴 때 토스트 집계, 필터 변경 시 스크롤 위치 보존, 필터 드롭다운의 바깥 클릭 닫힘을 추가했을 것입니다.
