# AI 도구 활용 (AI_USAGE)

## 1. 사용한 도구

- Claude Code (Opus 4.8) — 설계 논의와 설계 문서화, 그 설계에 따른 구현 주도, 테스트 케이스 설계

## 2. 주로 어디에 썼나

| 작업 | 사용한 도구 | 활용 방식 |
| ---- | ---- | ---- |
| 요구사항 분석과 아키텍처 설계 | Claude | 상태 구조 대안 3개(직접 구현 / TanStack Query / Zustand)를 트레이드오프 표로 비교. 최종 선택은 본인 |

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

1. **미측정 성능 수치 거부** — 설계 문서 초안에 "5,000개 전량 렌더는 초기 렌더 1.5초 이상(미측정이나 명백)"이라는 서술이 올라왔습니다. 측정하지 않은 수치를 사실처럼 쓰는 것을 거부하고, 스타터 기준선을 먼저 측정한 뒤 개선 후와 같은 조건에서 비교하는 절차로 바꿨습니다. DECISIONS 4절의 표가 그 결과이고, 둘 다 GitHub Pages 프로덕션 빌드에서 측정해 조건을 맞췄습니다.
2. **재조회 병합 방식 수정** — 재연결 후 GET 결과로 캐시를 "전체 교체"하는 안이 1차로 나왔습니다. 검토 중 PATCH 성공 응답과 GET 응답이 둘 다 진행 중일 때 더 오래된 GET 스냅샷이 나중에 도착하면 새 version을 덮어쓰는 시나리오를 확인했고, 태스크별 version 비교로 높은 쪽을 유지하는 병합(mergeServer)으로 바꿨습니다. `src/lib/mutations.test.ts`의 "캐시의 version이 높으면 오래된 fetched가 덮어쓰지 못한다" 테스트로 고정했습니다.
3. **구현 누락을 테스트가 적발** — 설계에는 "생성이 최종 실패하면 그 tempId를 참조하던 대기 뮤테이션을 연쇄 정리(orphan sweep)"가 있었는데, 엔진 1차 구현에는 이 정리가 빠져 있었습니다. 설계 문서의 케이스 매트릭스대로 테스트를 먼저 쓰면서 누락을 발견하고 엔진을 수정했습니다(커밋 "태스크 생성 - 모달과 tempId 낙관 삽입").
4. **jsdom에서 BroadcastChannel 테스트 가능하다는 가정 기각** — Node 18의 전역 BroadcastChannel로 다중 탭 동기화를 테스트할 수 있다는 접근이 제시됐지만, 실행해 보니 jsdom의 이벤트 클래스와 Node의 이벤트 클래스가 충돌해 메시지 전달이 깨졌습니다. 채널 계약(postMessage/onmessage/close)만 구현한 테스트 대체물로 바꾸고, 실브라우저 전파는 배포 URL에서 탭 2개로 수동 확인했습니다.

## 4. 스스로 판단해서 결정한 부분

- 상태 코어를 라이브러리 없이 직접 구현하기로 한 결정. AI는 대안별 장단점을 정리했고, "롤백 같은 순수 로직을 테스트로 증명하라"는 과제 요구와의 정합성을 기준으로 선택한 것은 본인입니다. 실무 경험이 있는 TanStack Query를 두고 내린 결정이라 그 이유를 DECISIONS 1절에 남겼습니다.
- 오프라인 정책(차단+재개), 409 해소(서버 우선+재적용), 재시도(자동 2회 후 수동)의 선택. 각 결정의 기각한 대안과 근거는 DECISIONS 5절에 있습니다.
- AI가 구현을 주도하는 방식 자체도 본인의 선택입니다. 그 전제로 모든 커밋을 검증하고, 코드 구조(파생 뷰, 직렬화, version 주입)를 설계 단계에서 함께 확정해 어느 부분이든 설명할 수 있는 상태를 유지했습니다.
